# MySQL 笔记

### MySQL 基础知识

#### MySQL 使用的树结构

- InnoDB 主流索引结构是 B+ 树。
- B+ 树非叶子节点主要存键和指针，叶子节点存数据或主键值，并通过链表有序连接。
- 适合数据库索引的原因：树高低、磁盘 I/O 少、查询稳定、范围查询效率高。
- 二叉树或普通平衡二叉树分叉少，数据量大时树高更高，不适合磁盘型数据库索引。

#### 索引优点

- 提升查询效率
- 降低排序成本，建立索引的时候是会选择排序的，默认 `ASC`（中文也可以排序）

#### 索引缺点

- 创建、维护索引需要耗费时间
- 索引需要占用物理存储空间，数据量越大，占用空间越大
- 会降低增删改的效率，因为 MySQL 需要动态维护索引

#### 适合使用索引的场景

- 较为频繁作为查询条件的字段应该创建索引

#### 不适合使用索引的场景

- 字段值唯一性太差（如年龄，很多相同）
- 更新非常频繁的字段不适合
- 不频繁作为查询条件的字段不合适

#### 索引种类

- 按数据结构分：B+ 树、Hash 索引
- 按物理存储分：聚簇索引、非聚簇索引
- 按字段特性分：主键索引、唯一索引、普通索引、全文索引
- 按字段个数分：单列索引、联合索引（又叫复合索引、组合索引）

#### B+ 树索引

- InnoDB 默认使用 B+ 树索引，适合等值查询、范围查询、排序和最左前缀匹配。

#### Hash 索引（索引列进行 hash 运算后取模，MySQL 用链表解决 hash 冲突问题）

- InnoDB 不支持用户显式创建普通 Hash 索引，但内部有自适应哈希索引优化热点访问。
- Memory 引擎才支持 hash 索引
- Hash 索引只能用于等值比较，比如：`name="张三"`，不支持范围查询，也不支持排序，因为 hash 是无序的

#### 聚簇索引 （可以简单理解为索引跟表数据在同一个文件下）

- InnoDB 表的数据按聚簇索引组织，聚簇索引的叶子节点保存整行数据。
- 一个表只能有一个聚簇索引，通常就是主键索引。
- 二级索引的叶子节点保存索引列和主键值，查询非索引列时可能需要回表。

#### 非聚簇索引 （可以简单理解为索引跟表数据分两个文件）

- 非聚簇索引的叶子节点不直接保存整行数据，通常保存定位行数据所需的指针或主键值。
- MyISAM 的索引和数据分离，索引叶子节点保存数据文件地址；InnoDB 的二级索引叶子节点保存主键值。

#### 单列索引

- 就是一个索引只有一个列，如设置为 `name`

#### 联合索引

- 就是一个索引有多个列，如设置为 `CREATE INDEX idx_name_age ON users (name, age);`
- 联合索引需要符合最左前缀原则，就是说 `WHERE` 后边一定要带上联合索引最左边的字段，如设置 (`name, age, city`) 为联合索引，那么 `WHERE` 一定要将 `name` 作为最开始的匹配条件
- 联合索引可以理解为纵排的单元格，都是有序的，索引顺序就是单元格顺序，如果第一位没有命中索引，第二位就无法检索到

| name | 张三 | 李四 |
| ---- | -- | -- |
| age  | 18 | 27 |
| city | 深圳 | 深圳 |

> 联合索引可以快速匹配到所需数据，大大减少扫描行数

#### 非聚簇索引（二级索引）

- 不是聚簇索引的索引叫非聚簇索引

#### 覆盖索引

- 使用非聚簇索引查询的时候，叶子节点中已经有了所需结果，无需再查询聚簇索引，这时候就是用到了覆盖索引

#### 回表查询

- 二次查询聚簇索引的行为就是回表查询

#### 索引下推

- 在索引遍历过程中，对索引中包含的字段先做判断，直接过滤掉不满足条件的记录
- 减少回表查询次数，提高查询效率

> `SELECT * FROM user WHERE name LIKE "张三%"`，其中 `id` 为主键，`name` 为非聚簇索引，由于没有覆盖索引，则需要回表，那么 5.6 之前，每拿到一个非聚簇索引数据就需要回表查询一次，而 5.6 之后是拿到所有非聚簇索引的指针后做一次性回表

#### 最左前缀原则

- 从第二列开始搜索不可能生效
- 即使查询条件没有完全按照索引的顺序设置，最左匹配原则仍然会作用于索引的前面部分，直到查询条件不再与索引匹配为止

```sql
-- 不生效，缺少最左的字段
SELECT *
FROM user
WHERE age = 19 AND city = '深圳';

-- name 生效，city 不生效，缺少中间字段
SELECT *
FROM user
WHERE name = '张三' AND city = '深圳';

-- 生效
SELECT *
FROM user
WHERE name = '张三' AND age = 18;
```

#### 索引失效

- 联合索引：不符合最左前缀原则
- 联合索引：范围之后全失效，`FORCE INDEX` (索引名称) 可强制使用索引，看实际业务
- 索引字段使用了函数、类型转换、计算、`OR`

```sql
-- name 生效，age 生效，city 失效
SELECT *
FROM user
WHERE name = '张三' AND age > 18 AND city = '深圳';

-- 索引失效，最左字段使用了范围
SELECT *
FROM user
WHERE name > '张三' AND age = 18 AND city = '深圳';

-- 不等于、IS NULL、OR 都可能影响优化器是否使用索引
SELECT *
FROM user
WHERE name <> '123';

SELECT *
FROM user
WHERE name IS NULL;

SELECT *
FROM user
WHERE name = '1' OR name = '2';
```

#### 雪花算法

- 用于在分布式系统中生成唯一的 64 位的序列化 ID
- ID 结构
	- 时间戳，长度 `41`，使用当前时间，时间戳的位数确保了 ID 按照时间顺序递增
	- 数据中心 ID，长度 `5`，用于标识分布式系统中不同的数据中心，最多可表示 `2^5` 个数据中心
	- 机器 ID，长度 `5`，用于标识数据中心中的不同机器，最多可表示 `2^5` 台机器
	- 序列号，长度 `12`，用于同一毫秒内生成不同的 ID，最多可以生成 `2^12` 不同的序列号
- 优点
	- 全局唯一性，生成的 ID 在分布式系统中是唯一的，不会出现重复
	- 有序性，生成的 ID 是按照时间顺序递增的，方便排序和索引
	- 高性能，算法简单，生成速度快，能够满足高并发场景的需求
- 注意事项
	- 时钟回拨，服务器的时间可能会回拨
- 并发控制，多线程或分布式生成时要保证同一时间戳下序列号递增且不重复

#### 如果一个表没有聚簇索引还会创建 B+ 树吗？

- 会。InnoDB 会优先选择主键作为聚簇索引；没有主键时选择第一个非空唯一索引；仍没有时生成隐藏的 `row_id` 作为聚簇索引。
- 因此建议为 InnoDB 表显式设置稳定、短小、递增或近似递增的主键。

#### 提升

- `SELECT *` 少用，需要什么字段写什么字段
- 数据过多可批量插入，每次最佳是 `1000` 条数据
- 连接的表不宜过多
- `LIKE` 使用后模糊，示例 `name LIKE "张%"`，不要使用后模糊
- 范围查询过大时要切割，此时优化器可能评估全表扫描更快

```sql
SELECT name 
FROM user 
WHERE age >= 2000 AND age <= 4000;

-- 优化
SELECT name 
FROM user 
WHERE age >= 2000 AND age <= 3000;

SELECT name 
FROM user 
WHERE age > 3000 AND age <= 4000;
```

- 小表驱动大表：`student` 表 30 条数据，`scores` 表有 10000 条数据，`FROM student LEFT JOIN scores` 就是小表驱动大表
- 用连接查询代替子查询，不绝对，视情况而定
- `GROUP BY` 列要加索引，示例，`GROUP BY name`，`name` 需要建立索引
- 一次查询数据过多一定要使用 `LIMIT`，`LIMIT` 越后查询越慢，这时候应该添加查询条件

```sql
-- 此时 MySQL 会先扫描并丢弃前 10000 行
SELECT id, name
FROM user
ORDER BY id
LIMIT 10000, 10;

-- 记住上一页最后一条 id 后再查询下一页
SELECT id, name
FROM user
WHERE id > 10000
ORDER BY id
LIMIT 10;

-- 使用覆盖索引先拿主键，再回表取完整行
SELECT u.*
FROM user AS u
JOIN (
  SELECT id
  FROM user
  ORDER BY id
  LIMIT 10000, 10
) AS page USING (id);
```

- 能用 `UNION ALL` 就不用 `UNION`，不用去重的话，一定要用 `UNION` 的话尽量用覆盖索引

### MySQL 基础架构

#### MySQL 底层架构

- Server 层
	- 连接器，管理连接和权限校验
	- 缓存，查询请求时，如果该查询在缓存中存在，并且对应的表数据没有发生变化，会直接返回缓存中的结果，MySQL 8.0 后，查询缓存功能已经被移除
	- 分析器，语法分析和词法分析
	- 优化器，优化语句和生成执行计划
	- 执行器，执行语句并返回结果
- 存储引擎层
	- MyISAM
	- InnoDB
	- Memory

#### InnoDB 和 MyISAM 的区别

|      | InnoDB   | MyISAM  |
| ---- | -------- | ------- |
| 事务   | 支持事务     | 不支持事务   |
| 文件   | 表空间文件保存数据和索引 | 数据文件和索引文件分离 |
| 主键   | 建议设置主键；不设置会生成隐藏行 ID | 可不设置主键 |
| 外键   | 支持外键     | 不支持外键   |
| 表总行数 | 不存储表的总行数 | 存储表的总行数 |
| 锁    | 支持表锁和行锁  | 仅支持表锁   |

- InnoDB 支持事务、行锁、外键，适合高并发和数据安全性比较高的场景
- MyISAM 提供高效存储和查询，适合读多和数据安全性较低的场景

#### 聚簇索引和非聚簇索引的区别

- 聚簇索引
	- 叶子节点保存了整行的数据
	- 行的物理存储顺序和索引顺序相同
	- 一个表只能有一个聚簇索引
	- 查询、排序更快
	- 插入、删除更慢
- 非聚簇索引
	- 叶子节点仅保存索引数据和主键
	- 行的物理存储顺序和索引顺序无关
	- 一个表可以有多个非聚簇索引，都是基于不同的列

#### 怎么创建联合索引的查询效率最高

- 区分度高的字段放前面，可减少查询次数，更适合创建索引

#### 创建索引的注意事项

- 避免创建过多索引
- 优先使用联合索引
- 使用联合索引时，区分度高的字段放前面
- 过长字符串可以使用前缀索引
- 值唯一的字段，使用唯一索引

#### 解决深分页问题

- 使用索引
- 分页优化，记住上次的值，下次从这个值开始

### MySQL 事务

#### MySQL 中事务的四大特性

- 原子性，事务要么成功要么失败
	- `undo log`
- 一致性，数据在事务提交前后始终处于一致性
	- 代码逻辑实现
- 隔离性，事务提交前的中间态不可见，是相互隔离的
	- MVCC
- 持久性，事务提交后数据的修改永远保存在数据库中
	- `redo log`
- 简称 ACID

> `redo log`，是物理日志，记录了磁盘数据页的修改，服务器崩溃后数据持久化到磁盘上
> `undo log`，是逻辑日志，记录了相反的 SQL 语句，回滚事务时恢复到之前的数据，实现了 MVCC
> `bin log`，是逻辑日志，记录了 SQL 语句，用于数据备份和同步
> MVCC，记录某个时间点的数据快照，用来实现不同事务之间数据的隔离性，MVCC 的实现方式通过两个隐藏列 `trx_id`（最近一次提交事务的 ID）和 `roll_pointer`（上个版本的地址），建立一个版本链

#### MySQL 并发事务产生的问题？

- **脏读**，一个事务读取到其他事务未提交的数据
- **幻读**，相同的查询条件，多次读取结果不一样，读到了其他事务 `INSERT` 提交后的数据
- **不可重复读**，相同的查询条件，多次读取结果不一样，读到了其他事务提交 `UPDATE`、`DELETE` 后的数据

#### MySQL 四大隔离级别的作用和问题

- **读未提交**，可读取事务未提交的数据，存在脏读、不可重复读、幻读的问题
- **读已提交**，可读取事务已提交的数据，存在不可重复读、幻读问题
- **可重复读**，相同的数据，多次读取的结果一致，存在幻读问题
- **串行化**，所有事务串行化，效率低

#### 什么是当前读和快照读

- 当前读，读取数据的最新版本，并对数据上锁，如 `UPDATE`、`INSERT`、`DELETE`
- 快照读，读取数据的历史版本，不对数据上锁，如 `SELECT`

#### MySQL 使用什么方法解决幻读

- 在当前读的情况下，对数据加锁解决幻读
- 在快照读的情况下，使用 MVCC 解决幻读

### MySQL 主从同步

#### MySQL 主从同步的作用及实现原理

- 读写分离，提升数据库性能
- 容灾恢复，主服务器不可用时，从服务器提供服务，提高可用性
- 冗余备份，主服务器数据损坏丢失，从服务器保留备份

> MySQL 主从复制基于 binlog。binlog 可采用 statement、row 或 mixed 格式；生产中常用 row 格式记录行变更，更利于复制一致性。

#### MySQL 主从同步延迟产生原因

- 从库机器性能较差
	- 主库负责所有读写请求，从库只用来备份，会用性能较差的机器，执行时间自然较慢
- 从库压力更大
	- 读写分离后，主库负责写请求，从库负责读请求
	- 互联网应用一般读请求更多，所以从库读压力更大，占用更多 CPU 资源
- 网络延迟
	- 当主库的 binlog 文件往从库发送时，可能产生网络延迟，也会导致从库数据跟不上
- 主库有大事务
	- 当主库上有大事务或大批量写入时，从库回放 binlog 也需要较长时间，容易产生复制延迟

####  MySQL 主从同步延迟的解决方案

- 从库机器性能较差
	- 把从库换成跟主库同等规格的机器
- 从库压力更大
	- 多搞几台从库，分担读请求压力
- 网络延迟
	- 联系运维或者云服务提供商解决
- 主库有大事务
	- 把大事务分割成小事务执行，大事务不但会产生从库延迟，还可能产生死锁，降低数据库并发性能，所以尽量少用大事务

### MySQL 分库分表

#### 什么情况需要分库、分表

- 当数据库的 QPS 过高，数据库连接数不足的时候，就需要分库
- 当单表数据量过大，读写性能较差，就需要分表
- 当两者都有的时候，就需要分库分表

> 分表能解决就尽量不分库

#### 如何实现分库分表，有哪些拆分方式

- 分库分表有垂直拆分和水平拆分，垂直拆分又有垂直分库、垂直分表
- **垂直分库**，不同的业务拆分到不同的数据库
- **垂直分表**，把长度较大或者访问频次较低的字段，拆分到扩展表中
- **水平分表**，单表数据量过大时，按照 ID 拆分到多张表中

#### 分库分表引入哪些问题

- **垂直分库：** 不同库多表之间无法 `JOIN` 关联查询，只能通过接口聚合，复杂度直线上升。横跨多个数据库导致无法使用本地事务，数据强一致性就别想了，只能引入更为复杂的分布式事务，勉强实现数据的最终一致性，可用性直线下降
- **垂直分表：** 本来一张表能查出来的数据，现在需要多张表 `JOIN` 关联查询，这不瞎耽误事
- **水平分表：** 多张表关联查询时，无法实现分页、排序功能

#### 分库分表引入的问题的解决方案

- 略

### MySQL 锁

#### 锁的分类

- 按照锁的粒度可分为，表锁、页面锁、行锁、记录锁、间隙锁、临键锁
- 按照锁的属性可以分为，共享锁、排他锁
- 按照锁的机制可以分为，乐观锁、悲观锁

#### 表锁

- 优点
	- 开销小、加锁快、不会出现死锁
- 缺点
	- 锁定粒度大、并发度低、锁冲突概率高
- 场景
	- 更新表中大部分数据
	- 事务涉及到多张表、业务逻辑复杂

#### 页面锁

- 优点
	- 开销、并发度、锁定粒度介于行锁和表锁之间
- 缺点
	- 会出现死锁

#### 行锁

- 优点
	- 并发度高、锁定粒度小、锁概率冲突低
- 缺点
	- 开销大、加锁慢、会出现死锁

#### 什么是记录锁（Record Locks）、间隙锁（Gap Locks）、临键锁（Next-Key Locks）

- 记录锁，对某条记录上锁

```sql
UPDATE user SET age = age + 1 WHERE id = 1;
```

- 间隙锁，对某个范围加锁，但是不包含范围的临界数据

```sql
UPDATE user SET age = age + 1 WHERE id > 1 AND id < 10;
```

- 临键锁，由记录锁和间隙锁组成，既包含本身又包含范围，左开右闭区间

```sql
UPDATE user SET age = age + 1 WHERE id > 1 AND id <= 10;
```

#### 什么是共享锁（又称读锁、S 锁）和排他锁（又称写锁、X 锁）

- 共享锁
	- 防止其他事务修改当前数据
	- 加锁方式，在 `SELECT` 语句末尾加上 `LOCK IN SHARE MODE` 关键字

```sql
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;
```

- 排他锁
	- 防止其他事务读取或者更新当前数据
	- 加锁方式，在 `SELECT` 语句末尾加上 `FOR UPDATE` 关键字

```sql
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

#### 什么是乐观锁和悲观锁，这些锁的作用及实现方式

- 乐观锁
	- 总是假设别人不会修改当前数据，所以每次读取数据的时候都不会加锁，只是在更新数据的时候通过 `version` 判断别人是否修改过数据
	- 适合读多写少场景

```sql
## 读取 version
SELECT id, name, age, version
FROM user
WHERE id = 1;

## 更新数据，判断 version 是否修改过
UPDATE user
SET age = age + 1,
    version = version + 1
WHERE id = 1 AND version = 1;
```

- 悲观锁
	- 总是假设别人会修改当前数据，所以每次读取的时候总是加锁
	- 适用于写多读少的场景

```sql
## 加读锁
SELECT *
FROM user
WHERE id = 1

## 加写锁
SELECT *
FROM user
WHERE id = 1
FOR UPDATE;
```

#### 不同索引之间的加锁范围解析

- MySQL 锁是加在索引记录上面的
- 如果是非唯一性索引，不论表中是否存在该记录，除了会对该记录所在范围加锁，还会向右遍历到不满足条件的范围进行加锁
- 如果是唯一索引，如果表中存在该记录，只对该行记录加锁。如果表中不存在该记录，除了会对该记录所在范围加锁，还会向右遍历到不满足条件的范围进行加锁
