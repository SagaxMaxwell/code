# 计算机网络面试笔记

#### 常见面试题

- [计算机网络](https://javaguide.cn/)

#### 多路复用

- 频分复用，将整个带宽分为多个频率不同的信道，每个信道传输一路信号
	- 电视与广播
- 时分复用，将传输时间分为不同的时间槽，每个时间槽传输一路信号的一部分
	- 电话
- 波分复用，主要用于光纤通信，它将不同波长的光信号合并到一根光纤中传输
	- 长距离光纤
- 码分复用，允许多个信号在同一频带上同时传输，每个信号被分配一个唯一的码序列
	- CDMA 手机网络
- 统计时分复用，基于时分复用，根据需求动态地分配时间槽，更加高效地利用信道容量
	- 以太网交换机

#### FTP

- 用于在网络上进行文件传输的标准网络协议，基于 TCP 协议，属于 C/S 架构，属于应用层协议
- 建立连接，客户端通过 TCP 协议与 FTP 服务器的 `21` 端口建立控制连接
- 用户认证，客户端提供用户名与密码，检验文件访问权限
- 选择传输模式
	- 用于传输文本文件使用 ASCII 模式
	- 传输非文本文件使用二进制模式（比特流）
- 数据连接，当需要进行文件传输时，FTP 会建立一个单独的数据连接，有两种方式
	- 主动模式，服务器主动向客户端发起数据连接
	- 被动模式，服务器打开一个端口并告知客户端，客户端主动连接到这个端口
- 文件传输
- 关闭连接

> FTP 明文传输用户名和密码，容易被窃听，为了提高安全性可使用 FTPS 或 SFTP
> FTPS，使用 SSL 或 TLS 加密 FTP 连接
> SFTP，使用 SSH 进行加密传输，与 FTP 不兼容

#### Socket

- Socket 是计算机网络中的一个概念，它提供了一种抽象层，允许应用程序通过网络进行通信
- Socket 类型
	- 流式套接字，基于 TCP 协议，提供可靠的字节流服务
	- 数据报套接字，基于 UDP 协议，提供不可靠的数据包服务
	- 原始套接字，提供较低层次的网络访问，可以用来发送和接收原始网络包
- 套接字编程步骤
	- 创建套接字，使用适当的套接字类型创建一个新的套接字对象
	- 绑定套接字，将套接字绑定到一个本地地址上，指定 IP 地址和端口号
	- 监听，对于服务器端套接字，监听指定的端口上的连接请求
	- 接受连接，对于服务器端套接字，接受一个来自客户端的连接
	- 连接套接字，对于客户端套接字，连接到一个远程地址
	- 发送和接收数据，通过套接字发送和接收数据
	- 关闭套接字，在数据传输完成后，关闭套接字

#### TCP 三次握手与四次挥手

- TCP 定义
	- 是面向连接的、可靠的、全双工的、流式传输的通信协议
- 三次握手握手建立连接
	- 客户端发送 SYN，进入 SYN_SENT 状态，等待服务器确认
	- 服务器收到 SYN 后返回 SYN-ACK，进入 SYN_RCVD 状态
	- 客户端收到 SYN-ACK 后向服务端发送 ACK 报文，将连接状态设置为 ESTABLISHED 状态，服务器收到 ACK 报文后，也将连接状态设置成 ESTABLISHED
- 为什么需要三次握手
	- 同步序列号
	- 三次握手的过程确保了双方都有接收和发送数据的能力
- 四次挥手释放连接
	- 客户端发送数据后，发送 FIN 到服务器，进入 FIN_WAIT_1 状态
	- 服务器收到 FIN 后，发送 ACK 报文作为应答，并将连接状态设置为 CLOSE_WAIT，客户端收到 ACK 后，进入 FIN_WAIT_2 状态
	- 服务器发送 FIN 给客户端，进入 LAST_ACK 状态
	- 客户端收到 FIN 报文后，发送 ACK 报文，进入 TIME_WAIT 状态，服务器收到 ACK 后设置成 CLOSE 状态并关闭连接
- 为什么会四次挥手
	- 半关闭状态，TCP 连接是全双工的，每个方向必须单独地关闭
	- 双方都要确认对方知晓终止连接的请求
	- 双方都要告知对方已经完成了数据的发送

> 客户端发送 FIN 表示它不再发送数据，但可以接收数据
> 服务器发送 FIN 表示它不再发送数据

#### TCP 与 UDP

| 特性    | TCP               | UDP                |
| ----- | ----------------- | ------------------ |
| 可靠性   | 提供可靠的数据传输         | 不提供可靠的数据传输         |
| 连接导向  | 面向连接，需要建立和释放连接    | 无连接，不需要建立和释放连接     |
| 拥塞控制  | 有拥塞控制机制           | 无拥塞控制，不考虑网络拥塞      |
| 流量控制  | 通过滑动窗口进行流量控制      | 无流量控制，发送速率由应用决定    |
| 数据包大小 | 受限于 MSS（最大分段大小）   | 受限于 MTU（最大传输单元）    |
| 传输速度  | 较慢，因为需要确认和重传      | 较快，因为没有确认和重传机制     |
| 错误检查  | 有校验和，检测传输错误并进行重传  | 有校验和，检测传输错误但不进行重传  |
| 应用场景  | 文件传输、电子邮件、Web 浏览等 | 直播、在线游戏、VoIP、视频会议等 |
| 头部开销  | 较大，因为需要维护连接状态     | 较小，因为不需要维护连接状态     |
| 状态    | 需要维护每个连接的状态信息     | 不需要维护连接状态信息        |

#### HTTPS 如何实现加密

- TCP 三次握手，客户端和服务器之间建立 TCP 连接
- SSL/TLS 握手
	- 客户端问候，客户端发送一个“客户端问候”消息，包含客户端支持的加密算法和随机数
	- 服务器问候，服务器响应一个“服务器问候”消息，选择一个加密算法，并发送服务器的证书（包含公钥）和另一个随机数
	- 密钥交换，客户端使用服务器的公钥生成一个随机密钥（预主密钥），并将其加密后发送给服务器
	- 生成会话密钥，服务器使用私钥解密接收到的预主密钥，双方使用预主密钥和之前的随机数生成会话密钥
- 加密通讯，双方使用会话密钥对通信内容进行加密和解密
- 数据传输，加密后的数据通过 TCP 连接传输

#### HTTP 和 HTTPS

- HTTP 是明文传输协议，HTTPS 协议是由 SSL + HTTP 协议构建的可进行加密传输、身份认证的网络协议，比 HTTP 协议安全
- HTTPS 标准端口 `443`，HTTP 标准端口 `80`
- HTTPS 需要用到 SSL 证书，而 HTTP 不用

> HTTP 基于 TCP

#### GET，POST，DELETE，PUT，PATCH

- GET
	- 用途：从服务器检索资源
	- 例子：`GET /items` 会从服务器获取 `/items` 路径下的资源
	- 特点：GET 方法不改变服务器上的资源，仅用于读取数据
- POST
	- 用途：向服务器发送数据，创建新资源
	- 例子：`POST /items` 会向服务器发送数据，创建一个新的 `/items` 资源
	- 特点：POST 方法可以发送大量数据，通常用于表单提交
- DELETE
	- 用途：从服务器删除资源
	- 例子：`DELETE /items/1` 会从服务器删除 `/items/1` 路径下的资源
	- 特点：DELETE 方法会永久删除服务器上的资源
- PUT
	- 用途：向服务器发送数据，替换现有资源
	- 例子：`PUT /items/1` 会向服务器发送数据，替换 `/items/1` 路径下的资源
	- 特点：PUT 方法可以发送大量数据，用于更新资源
- PATCH
	- 用途：向服务器发送数据，部分更新现有资源
	- 例子：`PATCH /items/1` 会向服务器发送数据，部分更新 `/items/1` 路径下的资源
	- 特点：PATCH 方法用于更新资源的部分内容，可以减少数据传输量

#### GET 与 POST

| 特性   | GET                  | POST                 |
| ---- | -------------------- | -------------------- |
| 数据传输 | 数据作为 URL 的一部分（查询字符串） | 数据作为请求体（body）发送      |
| 幂等性  | 是幂等的，多次执行不会改变服务器状态   | 不是幂等的，多次执行可能会改变服务器状态 |
| 缓存处理 | 浏览器可能会缓存 GET 请求的结果   | 浏览器不会缓存 POST 请求的结果   |
| 数据类型 | 通常只发送文本数据            | 可以发送任意类型的数据，包括文件     |
| 数据量  | 数据量通常较小              | 数据量可以很大              |

#### JWT

- JWT 是一种自包含令牌，通常包含用户标识、签发时间、过期时间和签名。
- 访问令牌应设置较短过期时间，刷新令牌可设置更长过期时间；具体时长由业务安全要求决定，没有统一默认值。
- JWT 签名只能防篡改，不能加密载荷，敏感信息不要直接放入 Payload。
- 需要主动失效令牌时，可用 Redis 维护黑名单、版本号或会话状态。

#### 短轮询

- 客户端按固定间隔请求服务器，服务器立即返回结果。
- 实现简单，但无数据更新时也会产生大量无效请求。
- 适合实时性要求不高、并发规模较小的场景。

#### 长轮询

- 客户端发起请求后，服务器在有新数据或超时后才返回。
- 相比短轮询减少空请求，实时性更好。
- 缺点是服务端需要维护挂起请求，高并发下要关注连接数、超时和资源占用。
- 更强实时场景通常考虑 WebSocket、SSE 或消息队列。

#### Cookie 和 Session

- Cookie
	- 存在于客户端，浏览器会按域名和路径自动携带。
	- 可通过 `Expires` 或 `Max-Age` 设置过期时间；不设置时通常是会话 Cookie。
	- 容量有限，不适合存储敏感数据或大量数据。
- Session
	- 通常指服务端会话：客户端只保存 Session ID，数据存储在服务端。
	- Flask 默认 Session 存储在客户端 Cookie 中，内容经过签名防篡改，但默认不是加密。
	- 服务端 Session 更适合保存较多或较敏感的会话数据，但需要服务端存储和清理机制。
	- Flask 的非永久 Session 默认随浏览器会话结束；永久 Session 的默认生命周期由 `PERMANENT_SESSION_LIFETIME` 配置控制。

#### XSS，CSRF，CORS

- XSS
	- 注入攻击，在目标网站上注入恶意脚本，用于窃取用户会话、个人信息
	- 应进行严格的输入验证
	- 使用 CSP 内容安全策略限制资源的加载和脚本执行
- CSRF
	- 跨站请求伪造，利用用户的权限迫使用户执行非本意的操作
	- 可使用 CSRF 令牌，并在每次表单提交时验证
	- 检查请求体头部（不绝对安全）
- CORS
	- 跨来源资源共享，允许资源被另一个域的网页所请求

#### NAT 地址转换协议

- 应用于内部网到外部网的转换
- 当数据包到达 NAT 路由器时，NAT 路由器会修改数据包的源 IP 地址和端口号，将其转换为路由器本身的公网 IP 地址和端口号
- 优点
	- NAT 不光可以缓解 IPv 4 地址资源短缺的问题
	- 隐藏内部网络的实际拓扑结构，使得外部网络无法直接访问内部网络中的设备，从而提高了内部网络的安全性
	- WAN 的 ISP 变更接口地址时，无需通告 LAN 内主机
	- LAN 内主机地址变更无需告知 WAN

#### 隧道 (Tunneling)

- 隧道是一种网络技术，它允许数据包通过一个网络被封装并在另一个网络中传输
- 原始数据包被封装在一个新的数据包中，这个新的数据包包含了原始数据包的头部和尾部信息，使得原始数据包能够在不同的网络之间传输

#### 直接序列路由（DSR）

- DSR 是一种无线自组织网络 (Wireless Ad Hoc Network) 中的路由协议
- 在 DSR 中，每个节点都具有路由功能，可以独立地发送和接收数据包
- 当一个节点需要发送数据包到另一个节点时，它会首先寻找一个可以直接到达目的节点的路径，如果找到路径，则直接发送数据包；如果没有找到路径，它会先将数据包发送到一个中继节点，然后中继节点再寻找一个可以到达目的节点的路径，并继续转发数据包，直到数据包到达目的节点

#### 隧道与 DSR

- 将隧道技术与 DSR 结合起来，可以实现跨网络的数据传输。在这种情况下，原始数据包被封装在一个新的数据包中，这个新的数据包包含了原始数据包的头部和尾部信息，以及 DSR 协议所需的路由信息。这样，数据包就可以通过 DSR 协议在无线自组织网络中传输，同时确保数据包能够在不同的网络之间传输
