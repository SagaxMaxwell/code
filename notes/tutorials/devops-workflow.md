# DevOps 与工作流

## 开发模型

| 模型 | 特点 | 适用场景 |
| --- | --- | --- |
| 瀑布模型 | 阶段固定、需求前置 | 需求稳定、合规文档要求高 |
| 敏捷开发 | 小步迭代、持续反馈 | 需求变化快、产品探索 |
| DevOps | 开发、测试、运维协作自动化 | 需要高频可靠交付 |

DevOps 不是某个工具，而是一套减少交付摩擦的协作方式：版本控制、自动化测试、持续集成、持续部署、可观测性、事故复盘。

## CI/CD

最小流水线：

1. 安装依赖。
2. 静态检查：Ruff、类型检查。
3. 单元测试。
4. 构建产物。
5. 集成测试或 E2E。
6. 部署到测试环境。
7. 人工或自动门禁后发布。

失败要尽早发生。格式、lint 和单元测试放前面，浏览器 E2E 放后面。

## E2E 自动化

E2E 只覆盖关键路径，例如登录、下单、支付回调、核心表单提交。数量太多会慢且脆弱。

稳定性原则：

- 使用语义定位或测试 ID。
- 每个测试独立准备数据。
- 避免依赖测试执行顺序。
- 失败时保留截图、trace、日志和网络请求。

## 测试开发流程

1. 读需求，识别风险。
2. 拆测试点：正常、异常、边界、权限、并发、兼容性。
3. 设计数据和环境。
4. 写自动化用例。
5. 接入 CI。
6. 缺陷回归和用例补充。

## AI 工作流工具

| 工具 | 定位 |
| --- | --- |
| Dify | LLM 应用平台，偏产品化、RAG、工作流 |
| LangGraph | 代码优先的状态图/Agent 编排 |
| LangFlow | 可视化构建 LLM 流程 |
| Gumloop | 面向业务自动化的可视化工作流 |

选型先看三个问题：

- 是否需要代码级控制和测试？需要则偏 LangGraph。
- 是否需要快速交付给非工程用户？可评估 Dify/Gumloop/LangFlow。
- 是否有私有化、审计、权限和数据隔离要求？
